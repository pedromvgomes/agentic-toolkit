## v0.3.1 — `agtk lock` actually works against real remotes

Patch release fixing a regression introduced by v0.3.0. The provider auto-split made `github.com/owner/repo/...` a parseable input, but the sourcestore handed the resulting schemeless URL straight to `git`, which read it as a filesystem path and failed every fetch with the misleading *"does not appear to be a git repository"*.

If you tried `agtk lock` against any real remote on v0.3.0, this is the release you want.

### What changed

- **`internal/sourcestore`**: a new `gitTransportURL` helper prepends `https://` to schemeless URLs before invoking `git ls-remote` / `git fetch`. Inputs that already specify a transport (`file://`, `ssh://`, scp-style `git@host:path`) are passed through unchanged. The cache key and lockfile entries keep the canonical scheme-less form, so lockfiles written under v0.3.0 (if any survived) and v0.3.1 are byte-identical.
- **`internal/cli`**: the multi-line error renderer that v0.3.0 added for `*stack.ParseError` now also handles arbitrary wrapped chains and `errors.Join` siblings. Resolver / provider / `git` failures render top-down with one labelled row per layer, with sibling failures separated by `— and —`. `runGit` returns `errors.Join(wrap, stderr)` so the actual git stderr lands on its own row instead of being concatenated onto the wrapper.

### Before

```
agtk: resolve: stack "<entry>" extends[0] ("github.com/owner/repo/..."): provider: ls-remote github.com/owner/repo.git: git ls-remote --symref github.com/owner/repo.git HEAD: exit status 128: fatal: 'github.com/owner/repo.git' does not appear to be a git repository
fatal: Could not read from remote repository.

Please make sure you have the correct access rights
and the repository exists.
```

…even when the repo absolutely does exist.

### After

```
diag: source github.com/owner/repo.git pulled in via stack extends from "<entry>"
wrote .agentic-toolkit.lock.yaml
```

For genuinely broken inputs:

```
agtk: command failed
  → resolve
  → stack "<entry>" extends[0] ("github.com/owner/missing/stacks/x.yaml")
  → provider
  → ls-remote github.com/owner/missing.git
  → git ls-remote --symref https://github.com/owner/missing.git HEAD
  → exit status 128
  → — and —
  → remote: Repository not found.
    fatal: repository 'https://github.com/owner/missing.git/' not found
```

### Compatibility

- No schema changes — stack manifests and lockfiles parse identically to v0.3.0.
- No CLI surface changes.
- Stderr layout for resolver/provider errors is now multi-line. Scripts that grep agtk's stderr for the old single-line shape will need to adjust.
