## v0.5.0 — `--config` flag for bare-repo + worktree workflows

agtk now accepts a global `--config <path>` flag that decouples *where the manifest lives* from *where the rendered output lands*. The flag was added for the bare-repo + typed-worktree layout managed by tools like [`gt`](https://github.com/pedromvgomes/gt), where agent sessions start at the bare-repo root (sibling to `.bare/`) but the actual `.agentic-toolkit.yaml` lives inside a worktree directory like `main/` or `feature/foo/`.

### Behavior

```
agtk --config main/.agentic-toolkit.yaml sync
```

- **Config read** from `main/.agentic-toolkit.yaml`.
- **Lockfile** written to `main/.agentic-toolkit.lock.yaml` — always next to the config.
- **Local `./...` refs** in the manifest resolve relative to `main/`.
- **Rendered output** (`.claude/`, `CLAUDE.md`, the per-platform manifest) lands in the **current working directory**, not in `main/`. So an agent session running from the bare-repo root sees its tools and rules at the root, even though the manifest lives in a worktree.

When `--config` is omitted everything collapses to the v0.4.x behavior: the manifest is `<cwd>/.agentic-toolkit.yaml`, the lockfile sits next to it, and renders write into cwd.

The flag works on every command: `lock`, `fetch`, `plan`, `render`, `sync`, `status`, `init`. For `init`, the supplied path is the file that gets scaffolded — parent directories are created if missing.

### Custom filenames

Because `--config` takes a *file path*, not a *directory*, you can also keep multiple stack files in the same repo:

```
agtk --config personal-stack.yaml lock
agtk --config team-stack.yaml lock
```

Each writes its own `.agentic-toolkit.lock.yaml` next to the chosen file. (The lockfile name is fixed; if you have multiple manifests in the same directory, give them different *directories*, or accept that they share a lockfile.)

### Why

Pre-v0.5, agtk treated cwd as both "where the manifest lives" and "where to write output." That was fine for single-worktree repos but broke the bare-repo workflow: the agent's cwd is the bare root, but no manifest lives there. The two concepts had to split.

### Compatibility

- No schema changes. Manifests and lockfiles parse identically to v0.4.x.
- Default behavior (no `--config`) is unchanged.
- New persistent flag `--config` is added on the root command. No other CLI surface changes.
