## v0.3.0 — friendlier stack-manifest errors

Two changes that turn the most common authoring mistake — a typo in `.agentic-toolkit.yaml` — from a wall-of-text error into a useful one. No schema changes; existing v0.2.x configs and lockfiles continue to work unmodified.

### Provider URL auto-split

Entries for the three big providers no longer require an explicit `.git/` boundary. The parser now infers it after `owner/repo` so users can paste the form they see in a browser:

```yaml
# all three of these now work:
skills:
  - github.com/owner/repo/agents/foo
  - bitbucket.org/team/proj/rules/style.md
  - codeberg.org/owner/repo/skills/bar@v1.2

# normalized internally to:
#   github.com/owner/repo.git/agents/foo
#   bitbucket.org/team/proj.git/rules/style.md
#   codeberg.org/owner/repo.git/skills/bar@v1.2
```

`gitlab.com` is intentionally excluded from the allowlist — nested groups (`gitlab.com/group/subgroup/repo`) make `owner/repo` segment-counting ambiguous, so gitlab users still need an explicit `.git/`. The same auto-split applies to `extends:` entries.

### Plain-language parse errors

Three things made the old error unreadable:

1. The file path was duplicated in the chain (`read <path>: <path>: invalid_entry: …`).
2. Every layer joined with `: `, producing one wall of text per failure.
3. The wording used internal jargon — *"skill names must be flat"* — that didn't tell you the actual restriction (no `/` in the name).

After:

```
agtk: failed to parse stack manifest
  file:    /tmp/agtk-test/.agentic-toolkit.yaml
  reason:  invalid_entry
  detail:  skills[0]: skill name "gitlab.com/group/repo/path/to/skill" looks like a URL but is missing the ".git/" boundary that separates the repo URL from the in-repo path. Add ".git/" after the repo name — for example: "gitlab.com/group/repo.git/path/to/skill"
```

The renderer kicks in for any error chain that unwraps to `*stack.ParseError`. Plain (non-parse) errors still render as the single-line `agtk: <message>` form.

### Why

A user reported the original error after writing this:

```yaml
skills:
  - github.com/pedromvgomes/gt/agentic/skills/use-gt
```

The old error said *"skill names must be flat (got "github.com/…/use-gt"); only commands accept nested namespacing"* — technically correct, but it took a parser-source dive to understand that the actual rule was "add `.git/` after the repo name" and that the diagnostic had nothing to do with namespacing in any user-facing sense.

Two pieces had the diagnostic info to do better and didn't use it: the validator could see the input started with a known host, and the CLI could see the wrapped error already carried the file path.

### Compatibility

- No stack-manifest schema changes — `.agentic-toolkit.yaml` and shareable stacks parse identically to v0.2.x.
- No lockfile schema changes.
- `*stack.ParseError.Error()` output is unchanged; only the top-level CLI renderer formats it across multiple lines. Programs that consume `agtk` stderr and grep for the old single-line shape will need to adjust.
- Bare names that previously failed (e.g. `github.com/owner/repo/...`) are now accepted as URLs. No previously-valid input becomes invalid.
