## v0.2.1 — `agtk update` refreshes shell completion

Fixes a gap that landed in v0.2.0: the new shell-completion install logic only ran in `install.sh`, so consumers who upgraded via `agtk update` ended up with a fresh binary but a stale (or missing) `_agtk` completion file. After this release, `agtk update` performs the same completion refresh the installer does, immediately after the in-place binary swap.

### Behavior

After a successful install, `agtk update` now:

- detects the user's shell from `$SHELL` (bash, zsh, fish — anything else is skipped with a one-line note),
- picks the same on-disk path `install.sh` would (`$(brew --prefix)/share/zsh/site-functions/_agtk` for brew zsh, `~/.zsh/completions/_agtk` otherwise; `~/.local/share/bash-completion/completions/agtk` for bash; `~/.config/fish/completions/agtk.fish` for fish),
- shells out to the just-replaced binary's `completion <shell>` subcommand so the script reflects the new command tree,
- prints the path it wrote and a one-line hint (e.g. "restart your shell to load completions").

Failures during the completion step are non-fatal — the binary update has already succeeded. A hint is printed to stderr suggesting `agtk completion <shell>` for a manual retry.

### Opting out

- `--no-completion` flag on `agtk update`.
- `AGTK_NO_COMPLETION=1` environment variable (same name `install.sh` honors).

### Why

`agtk update` swaps the binary atomically via `selfupdate.Apply` but had no post-install hook. The completion script lives outside the binary, so a binary upgrade alone never refreshed it. The fix lifts the path-resolution + write logic out of `install.sh` into `internal/completioninstall` and calls it from both entry points, keeping installer and updater in sync.

### Compatibility

No schema changes. Stack manifests, lockfiles, and the Claude render layout are identical to v0.2.0.
