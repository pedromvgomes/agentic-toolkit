## v0.2.0 — stack manifest model

Breaking schema bump. The v1 split between consumer config (`.agentic-toolkit.yaml` with `source:` + `presets:` + `externals:`) and preset bundles (`definitions/presets/*.yaml`) collapses into a single concept — the **stack manifest**. The consumer's entry-point file *is* a stack: same shape everywhere, with `extends:` for imports and per-category lists for definitions added on top.

If you're upgrading from v0.1.x, see [`docs/MIGRATION.md`](../MIGRATION.md) for the per-recipe translation table. Existing v1 configs produce a friendly `legacy_config` parse error pointing at that doc.

### Why

The v1 model had three pain points:

- **Custom bundles required publishing a preset.** A consumer who wanted to mix one external skill into the toolkit's `default` preset had to commit a preset file to the toolkit repo. Bundling-by-publishing made the toolkit responsible for every consumer's intent.
- **Presets only resolved against the primary source.** External preset references (`presets:` from another repo) weren't supported; only `externals:` for raw definition refs.
- **Layout was hardcoded.** Local definitions had to live under `definitions/<plural>/<name>...`. Consumers couldn't put their bundles wherever they wanted.

The stack model removes all three constraints: extend any repo's `stacks/*.yaml`, declare definitions inline (URL / `./path` / bare name), and place local bundles wherever you want via `root:` or path entries.

### Schema

```yaml
# .agentic-toolkit.yaml (or any stack file)
description: optional, only meaningful if this stack is imported
root: ./definitions          # optional; default for bare-name lookups

extends:                     # imported stacks — applied in declared order, later wins
  - github.com/owner/repo.git/stacks/default.yaml@ref
  - ./internal/team-stack.yaml

skills:                      # add definitions on top
  - challenge                # bare → resolved under <root>/skills/challenge/SKILL.md
  - ./local-skills/foo       # path → relative to this stack file's directory
  - github.com/owner/repo.git/skills/baz@ref   # external URL
agents:    [...]
rules:     [...]
instructions: [...]
commands:  [...]
hooks:     [...]
mcp:       [...]
settings:  [...]
```

Three-shape entry resolution:
- contains `.git/` → external URL ref
- starts with `./` or `/` → local path (relative to the stack file's directory)
- otherwise → bare name (resolved under `<root>/<plural>/<name>...` in the stack file's source FS)

`extends:` is processed depth-first, post-order: each imported stack's entries apply *before* the importing stack's own entries. Same `(category, name)` pair = later wins. The entry-point file's entries always win last. Cycle detection on the DAG.

### CLI changes

- **New `agtk sync`** — lock-if-stale + fetch + render in one shot. The two-pass primitives (`lock`, `fetch`, `render`) still exist for CI.
- **`agtk init --source` renamed to `--extends`** — seeds the first stack URL in the scaffolded extends list.
- **JSON output renames `preset` → `stack`** in `agtk plan --json` definition entries.

### Lockfile

Schema version bumped to **2**. v1 lockfiles are rejected with a migration hint pointing at `docs/MIGRATION.md`. Regenerate with:

```
rm .agentic-toolkit.lock.yaml
agtk lock     # or `agtk sync` to lock + fetch + render in one step
```

### Shareable stacks shipped in this release

- `stacks/default.yaml` — workflow-agnostic skills (`challenge`, `code-panel-review`, `pr-review-resolver`) plus the `plan-approval` instruction and the upstream `skill-creator` skill.
- `stacks/bare-repos.yaml` — bare-repo + worktree workflow rule. Stack on top of `default.yaml` when adopting that layout.

Reference them with the standard URL form:

```yaml
extends:
  - github.com/pedromvgomes/agentic-toolkit.git/stacks/default.yaml@v0.2.0
```

### Internals

- New `internal/sourceref` (shared `Source{URL, Ref}` type, dependency-free).
- New `internal/stack` (parser + types; one shape for entry-point files and shareable stacks).
- `internal/resolver` rewritten around extends-DAG traversal with cycle detection; `Plan.StackOrder` carries depth-first post-order visit order for adapter tiebreaking.
- `internal/config` and `internal/definitions/preset.go` deleted.
- `tools/schemagen` regenerates `definitions/SCHEMA.md` and `definitions/CONFIG-SCHEMA.md` for the new shape.

Net diff vs v0.1.x: ~2400 insertions / ~2280 deletions across 77 files — the simplification removes more than it adds.

### Also in this release

Three install-related changes that landed alongside the schema work:

- **Platform-detecting installer** (#14) — `install.sh` downloads the right release archive for your OS/arch, verifies its sha256 against `checksums.txt`, and drops `agtk` on `$PATH` without a Go toolchain.
- **Shell completion install** (#15) — the installer now writes shell completion for `bash`, `zsh`, or `fish` based on `$SHELL`. Opt out with `AGTK_NO_COMPLETION=1`.
- **Zsh completion path fix** (#16) — completion is written to a path the shell actually loads from (brew's `share/zsh/site-functions/` for zsh-on-brew users; `~/.zsh/completions/` with an `fpath` hint otherwise).

### Compatibility

- **Breaking** for consumer `.agentic-toolkit.yaml` and lockfiles. The CLI binary itself updates in place via `agtk update`; consumer files need a one-time rewrite per `docs/MIGRATION.md`.
- The Claude Code render layout (`.claude/skills/...`, managed-region `CLAUDE.md`, `_meta.agtk.managed` settings.json marker) is unchanged. Existing manifests, settings, and instructions files remain valid.
- The leaf definition schema (skill / agent / command / rule / instruction / hook / mcp / setting frontmatter) is unchanged.
