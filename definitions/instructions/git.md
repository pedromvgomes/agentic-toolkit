## Interacting with Git

When interacting with Git — especially creating commits and opening PRs — you MUST always follow these rules.

### 1. Determine the repo layout first

Before any commit, push, or PR operation, check whether you are in a `gt`-powered bare repo:

- The repo root contains a `.bare/` subfolder (the actual bare git repository), and branches/worktrees are checked out as sibling subfolders under that same root.
- If this layout is present, proceed with the rules below.
- If it is NOT present (this is a normal, non-bare repo), STOP and ask the user how they want Git authorship and authentication handled, rather than assuming.

### 2. Operate as the correct GitHub user

In a `gt`-powered bare repo, the root `.envrc` (auto-generated by `gt set-auth`) scopes `GH_TOKEN` to a specific GitHub user via a line like `gh auth token --user <username>`.
Read the `<username>` from that `--user` argument before running any `gh` command. If `.envrc` is missing or you cannot determine the user, ask and store the answer in memory
rather than guessing or falling back to a default.

### 3. Commit and PR hygiene

- **Never add co-authorship trailers.** Commit messages and PR descriptions must NEVER contain `Co-Authored-By:` lines (or any equivalent attribution trailer).
- **Never merge a PR** unless explicitly told to.
- **Always include a "Merge Commit Message" section** in every PR description — a short, descriptive message suitable for use as the commit message when the PR is squashed/merged.
